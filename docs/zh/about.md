# 关于 TrueNeverStory — 设计 rationale

## 语言策略：English inside, translate at boundary

### 为什么使用英语进行代理处理

TrueNeverStory 使用 **"English inside, translate at boundary"** 架构，原因如下：

1. **LLM 质量** — 大型语言模型在英语（其主要训练语言）上表现最佳。使用英语进行内部处理可确保：
   - 更一致的叙事质量
   - 更好地理解复杂提示
   - 更少的幻觉和风格不一致
   - 访问所有文学参考

2. **Token 经济性** — 英语通常比其他语言在相同语义内容上 token 效率高 20-40%。这意味着：
   - 更多上下文可以放入提示窗口
   - 每个请求的 API 成本更低
   - 处理时间更快

3. **文学丰富性** — 源材料（圣经、古腾堡经典）主要是英语或有规范的英语翻译。使用英语处理可保留：
   - 直接访问原型模式
   - 源文本的风格真实性
   - 细致的情感和主题内容

### 翻译管道

```
用户输入（任何语言）
    ↓
TranslationService.translateToEnglish()
    ↓
意图解析（英语）
    ↓
代理处理（英语）
    ↓
响应生成（英语）
    ↓
TranslationService.translate()
    ↓
用户输出（原始语言）
```

**关键设计决策：**
- 翻译在**输入时发生一次**，在**输出时发生一次**
- 所有内部状态、记忆和处理保持英语
- 代理从不直接看到或产生非英语文本
- UI 翻译与内容翻译分开（i18n vs TranslationService）

## 文学数据库架构：通过预处理实现 token 经济性

### 问题

从零开始生成丰富、文学性的叙述需要：
- 带有风格示例的大型提示
- 多次 LLM 调用以处理不同方面（情节、风格、情感）
- 高 token 消耗以获得高质量结果

### 解决方案：离线文学编译

TrueNeverStory 在**部署前**将文学源预处理为结构化 SQLite 数据库：

```
源文本 → LiteraryCompiler → SQLite 数据库 → 运行时查询
     ↓              ↓                    ↓              ↓
  Bible.db    4 遍解析器          FTS5 索引        毫秒查询
  Gutenberg   （剧作、            任务模板
  经典         风格、             风格模式
               情感、
               元数据）
```

### 数据库类型

| 数据库 | 来源 | 内容 | 用途 |
|--------|------|------|------|
| `bible.db` | 圣经文本 | 任务模板、原型、道德困境 | 叙事结构 |
| `gutenberg.db` | 古腾堡计划 | 风格模式、感官描述、节奏 | 文学质量 |
| `literary.db` | 编译输出 | 带 FTS5 搜索的统一模板 | 运行时访问 |

### Token 节省

**无预处理：**
```
提示："以古代史诗文学风格生成一个关于背叛的任务..."
Token：提示约 500-800 + 响应约 300-500 = 约 800-1300 token
```

**有预处理：**
```
查询：db.queryTemplates({ archetype: 'betrayal', mood: 'epic' })
Token：查询约 50 + 响应约 200-300 = 约 250-350 token
```

**节省：每个叙事元素 token 使用量减少 60-75%。**

## 丰富的文学来源

### 圣经原型

圣经提供了**经过时间考验的叙事结构**，在不同文化中产生共鸣：

| 原型 | 来源 | 模式 | 现代应用 |
|------|------|------|----------|
| **逃脱** | 出埃及记 14 | 领袖 → 暴君 → 障碍 → 干预 → 自由 | 叛乱任务、逃脱场景 |
| **审判** | 列王纪上 3 | 争端 → 明智统治者 → 隐藏真相 → 正义 | 宫廷阴谋、道德困境 |
| **继承** | 路加福音 15 | 浪子 → 浪费 → 回归 → 接纳 | 救赎弧线、家庭戏剧 |
| **起落起** | 创世记 37-50 | 受宠 → 被背叛 → 苦难 → 崛起 → 和解 | 角色发展弧线 |
| **忍耐** | 约伯记 | 苦难 → 怀疑 → 坚持 → 恢复 | 考验玩家决心 |
| **解放** | 士师记 | 压迫 → 呼召 → 聚集 → 胜利 | 战争战役、革命故事 |

**为什么圣经模式有效：**
- **普遍认知** — 玩家直觉地理解这些结构
- **道德复杂性** — 圣经叙事很少有简单的善恶划分
- **情感深度** — 失去、希望、背叛、救赎的主题
- **可扩展的戏剧** — 适用于亲密故事和史诗战役

### 古腾堡风格模式

古腾堡计划提供了**几个世纪的文学工艺**：

| 时代 | 作者 | 风格元素 | 用例 |
|------|------|----------|------|
| **哥特式** | 爱伦·坡、雪莱、斯托克 | 黑暗氛围、感官恐惧、心理紧张 | 恐怖、悬疑 |
| **维多利亚时代** | 狄更斯、勃朗特姐妹 | 社会评论、详细描述、道德复杂性 | 社会阴谋 |
| **史诗** | 荷马、弥尔顿 | 宏大规模、英雄语言、神话共鸣 | 战争、任务 |
| **浪漫主义** | 拜伦、济慈 | 情感强度、自然意象、激情 | 爱情故事、个人戏剧 |

**去词汇化过程：**
1. 提取结构模式（句子长度、节奏、词汇）
2. 移除角色名称和特定引用
3. 保留感官标记和情感基调
4. 创建带变量的可重用模板

## NPC 个性与特征

### 多层角色系统

每个 NPC 有**四个深度层次**：

```
L1：基本信息（姓名、角色、位置）
    ↓
L2：个性（特质、怪癖、说话模式）
    ↓
L3：隐藏动机（秘密目标、恐惧、欲望）
    ↓
L4：动态状态（关系、记忆、情感状态）
```

### 角色来源

| 来源 | 贡献 | 示例 |
|------|------|------|
| **圣经原型** | 道德框架、忠诚模式 | 路得的忠诚、大卫的野心 |
| **古腾堡角色** | 说话模式、社会行为 | 狄更斯的社会攀爬者、勃朗特姐妹的热情灵魂 |
| **历史模式** | 政治行为、派系动态 | 宫廷阴谋、行会政治 |
| **心理学模型** | 个性一致性、情感反应 | 大五人格特质、依恋风格 |

### NPC 记忆系统

```
短期：最近 3 次互动（ immediate context）
    ↓
中期：重要事件（关系变化）
    ↓
长期：核心记忆（形成性经历）
    ↓
语义：基于嵌入的回忆（上下文记忆）
```

**记忆影响：**
- 对话语调和词汇
- 信任水平和帮助意愿
- 问候风格（热情、冷淡、害怕）
- 话题可用性（个人、派系、任务）

### 经济和社会行为

NPC 具有基于历史模式的**现实经济行为**：

| 行为 | 来源 | 实现 |
|------|------|------|
| **贸易** | 中世纪商人行会 | 供需、基于声誉的定价 |
| **工艺** | 历史工匠作坊 | 技能水平、材料质量、时间投入 |
| **社会动态** | 封建等级 | 领主/附庸关系、派系忠诚 |
| **政治阴谋** | 宫廷政治 | 秘密联盟、信息交易 |

## 情节转折机制

### 叙事模式选择

当玩家采取行动时，系统会：

1. **分析意图** — 玩家试图做什么？
2. **模拟结果** — 实际会发生什么？
3. **选择原型** — 哪种圣经/文学模式合适？
4. **应用风格** — 哪种文学时代/情绪匹配？
5. **生成散文** — 结合模式 + 风格 + 上下文

### 示例：玩家背叛盟友

```
意图：背叛
模拟：关系被破坏，派系紧张加剧
原型：创世记 37（约瑟夫被兄弟出卖）
风格：维多利亚时代社会戏剧
结果：以狄更斯式的社会细节探索背叛后果的叙事
```

### 动态难度

任务模板包含**道德模糊性分数**（0-1）：
- 0.0 — 清晰的善恶（儿童故事）
- 0.5 — 复杂选择（标准 RPG）
- 1.0 — 没有正确答案（成熟叙事）

## 性能架构

### 为什么预处理获胜

| 方法 | 延迟 | Token 成本 | 质量 |
|------|------|-----------|------|
| **即时 LLM** | 2-5 秒 | 高 | 可变 |
| **预处理数据库** | <100ms | 最小 | 一致 |
| **混合（TNS）** | <100ms + 抛光 | 低 | 高 |

### 混合方法

1. **数据库查询** — 获取结构化模板（毫秒）
2. **变量替换** — 填充上下文（微秒）
3. **风格应用** — 应用文学模式（毫秒）
4. **LLM 抛光** — 最终散文生成（可选，用于关键时刻）

这给了我们**数据库速度**与**LLM 质量**在重要的地方。

## 世界规则与经济

### 封建等级

TrueNeverStory 实现了一个**10 级封建等级系统**，管理财富、税收、特权和社会互动：

| 等级 | 最小财富 | 守卫 | 基础税率 | 薪水 | 可以行贿 | 可以受贿 |
|------|----------|------|----------|------|----------|----------|
| **奴隶** | 0 | 0 | 100% | 0 | 否 | 否 |
| **平民** | 0 | 0 | 90% | 0 | 是 | 否 |
| **准男爵** | 100,000 | 50 | 30% | 0 | 是 | 是 |
| **男爵** | 500,000 | 200 | 28% | 0 | 是 | 是 |
| **子爵** | 2,000,000 | 1,000 | 25% | 0 | 是 | 是 |
| **伯爵** | 10,000,000 | 5,000 | 22% | 0 | 是 | 是 |
| **侯爵** | 50,000,000 | 20,000 | 20% | 0 | 是 | 是 |
| **公爵** | 200,000,000 | 100,000 | 18% | 0 | 是 | 是 |
| **国王** | 1,000,000,000 | 500,000 | 15% | 0 | 是 | 是 |
| **皇帝** | 10,000,000,000 | 2,000,000 | 10% | 0 | 是 | 是 |

**关键规则：**
- **奴隶**不能参与贿赂经济（没有自由意志）
- **平民**可以行贿但不能受贿（没有权力）
- **更高等级**支付更低的税收（权力折扣）但维护成本更高
- **财富阈值**必须达到才能晋升

### 税收系统

税收系统是**动态的**，受多种因素影响：

```
有效税率 = 基础税率 × (1 - 权力折扣 - 声望折扣)
```

**组成部分：**
- **基础税率** — 由等级定义（平民 90%，皇帝 10%）
- **权力折扣** — 基于政治权力最多减少 90%（权力 / 10,000）
- **声望折扣** — 基于公众认可最多减少 30%（声望 / 3,000）

**税收流动：**
```
NPC 收入 → 税收计算 → 国库征收 → 领主国库
     ↓          ↓          ↓            ↓
  基于等级    动态税率    每回合处理    沿封建链
  薪水                      带忠诚检查    累积
```

**高税收后果：**
- **背叛风险** = (税收负担 + 贿赂) / 收入 × (1 - 忠诚 / 1000)
- 高税收负担增加叛乱可能性
- 忠诚作为对抗背叛的缓冲

### 贿赂经济

贿赂是影响政治、忠诚和权力动态的**一流经济机制**：

**贿赂类型：**
| 类型 | 目的 | 风险等级 |
|------|------|----------|
| **保护** | 避免惩罚或迫害 | 中等 |
| **好意** | 获得优待 | 低 |
| **沉默** | 隐藏秘密 | 高 |
| **访问** | 到达受限区域或人员 | 中等 |
| **晋升** | 提升等级或职位 | 高 |
| **豁免** | 避免税收或义务 | 非常高 |

**贿赂风险公式：**
```
风险 = 基础风险 (10%) + 金额风险 (金额 / 10,000) + 目击者风险 (目击者 × 15%) - 接收者技能 (谋略 × 0.1%)
```

**风险因素：**
- **金额** — 更大的贿赂风险更高
- **目击者** — 更多观察者增加发现几率
- **接收者的谋略** — 熟练的官员可以隐藏腐败
- **贿赂类型** — 某些类型本质上风险更高

**经济影响：**
- 贿赂**降低忠诚**（腐败侵蚀信任）
- 贿赂**增加接收者的财富**
- 贿赂**增加权力**（贿赂金额的 1% 转化为权力）
- 贿赂**降低声望**（公众不赞成腐败）

### 经济周期（约瑟夫模型）

基于**圣经的约瑟夫模型**（创世记 41），经济循环经历三个阶段：

**阶段循环：**
```
丰裕 (30天) → 过渡 (10天) → 饥荒 (20天) → 丰裕...
```

**阶段效果：**
| 阶段 | 价格修正 | 储备变化 | 叙事事件 |
|------|----------|----------|----------|
| **丰裕** | 0.8× (便宜) | 每回合 +20% | 收获、贸易繁荣 |
| **过渡** | 1.0× (正常) | 稳定 | 市场不确定性 |
| **饥荒** | 2.0× (昂贵) | 每回合 -30% | 干旱、瘟疫、战争 |

**战略意义：**
- **在丰裕期间储存储备**以在饥荒中生存
- **价格规划**需要预见阶段转换
- **玩家决定**可以影响阶段持续时间和严重程度

### 禧年制度

每**50年**，**禧年事件**触发大规模经济重置：

**禧年效果：**
1. **债务免除** — 所有未偿还债务被免除
2. **土地归还** — 所有抵押土地归还原主
3. **忠诚加成** — 所有 NPC +30% 忠诚加成（持续 10 天）

**历史基础：**
基于**圣经的禧年**（利未记 25），防止永久贫困并维持社会流动性。

**战略意义：**
- **债务时机** — 避免在禧年附近借贷
- **土地获取** — 临时所有权创造有趣的动态
- **社会稳定** — 禧年防止极端财富集中

### 派系税收困境

当两个派系有冲突的税收主张时，系统为玩家生成**道德困境**：

**困境生成：**
- 每个经济回合 30% 几率
- 困境间最少 30 天冷却
- 税收金额从 100 到 1,000 金币

**玩家选择：**
| 选择 | 派系 A 忠诚 | 派系 B 忠诚 | 声望 |
|------|------------|------------|------|
| **支付派系 A** | +50 | -30 | 中立 |
| **支付派系 B** | -30 | +50 | 中立 |
| **拒绝两者** | -20 | -20 | -10 (不可靠) |

**叙事整合：**
每个困境生成解释冲突的上下文故事，迫使玩家做出有意义的政治选择。

### 派系劳动规则

每个派系可以设置**个别劳动政策**，影响工资和忠诚：

**劳动规则参数：**
- **固定工资** — 是/否（可预测 vs. 基于绩效）
- **工资金额** — 每个工作期间的基本工资
- **忠诚修正** — 工人忠诚的奖励/惩罚

**工资计算：**
```
最终工资 = 基本工资 × 工作小时数 × 生产率 × 派系修正
```

**忠诚冲突：**
当 NPC 为多个派系工作时，可能出现忠诚冲突：
- **双重忠诚**降低效率
- **派系转换**有声誉成本
- **工作停止**在冲突期间发生

### 奴隶经济

反映历史权力动态的**道德上复杂的**系统：

**奴隶属性：**
- **健康** — 身体状况（影响价值）
- **经验** — 技能和知识（增加价值）
- **恶习** — 贪婪、愤怒、懒惰（降低价值）
- **家庭** — 奴隶可以有家庭（创造义务）

**奴隶价值公式：**
```
价值 = 100 + (经验 × 0.1) + (健康 × 0.05) - 恶习惩罚
```

**奴隶生命周期：**
1. **奴役** — 通过债务、捕获或出生
2. **劳动** — 生产货物（每回合 300-1000 单位）
3. **叛乱** — 如果守卫薄弱则可能（力量比率决定成功）
4. **解放** — 花费 200 金币，授予平民身份

**伦理考虑：**
- 系统设计为**道德上不舒服**
- 玩家选择对奴隶 NPC 有**实际后果**
- **叛乱机制**确保奴隶制永远不会"安全"
- **解放路径**提供道德救赎机会

### 与社交图谱整合

经济与**社交图谱**整合，实现真实的权力动态：

**封建关系：**
```
封臣 → 领主 (税收 + 军事义务)
    ↓
指挥链 (多个层级)
    ↓
税收流动累积 (征收的税收)
```

**派系经济：**
- **经济派系**控制贸易和资源
- **军事派系**提供安全（以换取报酬）
- **宗教派系**影响道德选择
- **犯罪派系**在合法经济之外运作

**声望系统：**
- **公众认可**影响税率
- **派系地位**决定资源获取
- **个人声誉**影响贿赂成功率

### 经济数据库

所有经济数据存储在专用的 **SQLite 数据库**（`economic.db`）中：

**表：**
- `economic_cycles` — 带价格修正的阶段跟踪
- `jubilee_events` — 禧年历史记录
- `faction_labor_rules` — 每派系工资政策
- `faction_dilemmas` — 税收争议历史

**性能：**
- **索引查询**用于快速经济计算
- **批量处理**用于 NPC 操作（年龄衰减、恶习衰减、税收、忠诚）
- **Mojo 内核**用于计算密集型操作（5× 加速）

## 总结

TrueNeverStory 的架构结合了：

1. **英语优先处理** 以获得 LLM 质量和 token 效率
2. **预处理文学数据库** 以即时访问丰富的叙事模式
3. **多层 NPC 系统** 以获得可信、难忘的角色
4. **混合生成** 平衡速度和质量

结果是一个能够以**数据库速度**生成**文学质量叙事**的系统，同时保持**深度角色一致性**和**有意义的玩家选择**。
