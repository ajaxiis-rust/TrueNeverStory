# TrueNeverStoryについて — 設計の根拠

## 言語戦略：English inside, translate at boundary

### なぜエージェント処理に英語を使うのか

TrueNeverStoryは、いくつかの重要な理由から**「English inside, translate at boundary」**アーキテクチャを採用しています：

1. **LLM品質** — 大規模言語モデルは、主要な訓練言語である英語で最も良く機能します。内部処理に英語を使用することで、以下が保証されます：
   - より一貫した物語品質
   - 複雑なプロンプトのより良い理解
   - 幻覚やスタイルの不一致が少ない
   - 文学的参照の全範囲へのアクセス

2. **トークン経済** — 英語は同じ意味内容に対して他の言語より通常20-40%トークン効率が良いです。这意味着：
   - プロンプトウィンドウにより多くのコンテキストが収まる
   - リクエストあたりのAPIコストが低い
   - 処理時間が速い

3. **文学的豊かさ** — ソース資料（聖書、グーテンベルクの古典）は主に英語か、英語の正典翻訳があります。英語での処理は以下を保存します：
   - アーキタイプパターンへの直接アクセス
   - ソーステキストからのスタイルの真正性
   - 紋様的な感情的・テーマ的内容

### 翻訳パイプライン

```
ユーザー入力（任意の言語）
    ↓
TranslationService.translateToEnglish()
    ↓
意図解析（英語）
    ↓
エージェント処理（英語）
    ↓
応答生成（英語）
    ↓
TranslationService.translate()
    ↓
ユーザー出力（元の言語）
```

**重要な設計決定：**
- 翻訳は**入力時に1回**と**出力時に1回**行われる
- すべての内部状態、メモリ、処理は英語のまま
- エージェントは非英語テキストを直接見たり作成したりしない
- UI翻訳はコンテンツ翻訳とは別（i18n vs TranslationService）

## 文学データベースアーキテクチャ：前処理によるトークン経済

### 問題

ゼロからリッチで文学的な物語を生成するには、以下が必要です：
- スタイル例を含む大きなプロンプト
- 異なる側面（プロット、スタイル、感情）のための複数のLLM呼び出し
- 品質の高い結果のための高いトークン消費

### 解決策：オフライン文学コンパイル

TrueNeverStoryは、**デプロイ前に**文学ソースを構造化されたSQLiteデータベースに前処理します：

```
ソーステキスト → LiteraryCompiler → SQLiteデータベース → ランタイムクエリ
     ↓              ↓                    ↓              ↓
  Bible.db    4パスパーサー        FTS5インデックス    ミリ秒クエリ
  Gutenberg   （劇構成的、         クエストテンプレート
  古典        スタイル的、         スタイルパターン
              感情的、
              メタデータ）
```

### データベースタイプ

| データベース | ソース | 内容 | 目的 |
|-------------|--------|------|------|
| `bible.db` | 聖書テキスト | クエストテンプレート、アーキタイプ、道徳的ジレンマ | 物語構造 |
| `gutenberg.db` | Project Gutenberg | スタイルパターン、感覚的描写、ペース | 文学的品質 |
| `literary.db` | コンパイル出力 | FTS5検索付き統合テンプレート | ランタイムアクセス |

### トークン節約

**前処理なし：**
```
プロンプト：「古代叙事詩のスタイルで裏切りについてのクエストを生成...」
トークン：プロンプトに~500-800 + 応答に~300-500 = ~800-1300トークン
```

**前処理あり：**
```
クエリ：db.queryTemplates({ archetype: 'betrayal', mood: 'epic' })
トークン：クエリに~50 + 応答に~200-300 = ~250-350トークン
```

**節約：物語要素あたりのトークン使用量を60-75%削減。**

## 豊かな文学的ソース

### 聖書のアーキタイプ

聖書は、文化を超えて共鳴する**時間によって検証された物語構造**を提供します：

| アーキタイプ | ソース | パターン | 現代の応用 |
|-------------|--------|---------|------------|
| **脱出** | 出エジプト記14 | リーダー → 暴君 → 障害 → 介入 → 自由 | 反乱クエスト、脱出シナリオ |
| **裁き** | 列王記上3 | 争い → 賢い支配者 → 隠された真実 → 正義 | 宮廷の陰謀、道徳的ジレンマ |
| **相続** | ルカ15 | 放蕩 → 浪費 → 帰還 → 受容 | 贖罪のアーク、家族ドラマ |
| **栄転・転落・再起** | 創世記37-50 | 寵愛 → 裏切り → 苦難 → 栄転 → 和解 | キャラクター発展アーク |
| **忍耐** | ヨブ記 | 苦難 → 疑い → 持続 → 回復 | プレイヤーの決意を試す |
| **解放** | 士師記 | 抑圧 → 召命 → 集結 → 勝利 | 戦争キャンペーン、革命の物語 |

**聖書パターンが機能する理由：**
- **普遍的な認識** — プレイヤーはこれらの構造を直感的に理解する
- **道徳的複雑さ** — 聖書の物語は単純な善悪の区分が少ない
- **感情的な深さ** — 喪失、希望、裏切り、贖罪のテーマ
- **スケーラブルなドラマ** — 親密な物語と壮大なキャンペーンの両方に機能する

### グーテンベルクのスタイルパターン

Project Gutenbergは**何世紀もの文学的職人技**を提供します：

| 時代 | 著者 | スタイル要素 | ユースケース |
|------|------|-------------|-------------|
| **ゴシック** | ポー、シェリー、ストーカー | 暗い雰囲気、感覚的恐怖、心理的緊張 | ホラー、ミステリー |
| **ヴィクトリア朝** | ディケンズ、ブロンテ姉妹 | 社会批評、詳細な描写、道徳的複雑さ | 社会的陰謀 |
| **叙事詩** | ホーマー、ミルトン | 大規模、英雄的言語、神話的共鳴 | 戦争、クエスト |
| **ロマン主義** | バイロン、キーツ | 感情的強度、自然のイメージ、情愛 | 恋愛物語、個人的なドラマ |

**脱語彙化プロセス：**
1. 構造パターンを抽出（文長、リズム、語彙）
2. キャラクター名と特定の参照を除去
3. 感覚マーカーと感情的トーンを保存
4. 変数付きの再利用可能なテンプレートを作成

## NPCの個性とキャラクター

### 多層キャラクターシステム

各NPCは**4つの深さレベル**を持っています：

```
L1: 基本情報（名前、役割、場所）
    ↓
L2: 性格（特性、癖、 speech patterns）
    ↓
L3: 隠された動機（秘密の目標、恐れ、欲望）
    ↓
L4: 動的状態（関係、記憶、感情状態）
```

### キャラクターソース

| ソース | 貢献 | 例 |
|--------|------|-----|
| **聖書のアーキタイプ** | 道徳的フレームワーク、忠誠パターン | ルツの忠誠、ダビデの野心 |
| **グーテンベルクのキャラクター** | speech patterns、社会的行動 | ディケンズの社会的登り龍、ブロンテ姉妹の情熱的な魂 |
| **歴史的パターン** | 政治的行動、派閥力学 | 宮廷の陰謀、ギルド政治 |
| **心理学的モデル** | 性格の一貫性、感情的反応 | ビッグファイブ特性、愛着スタイル |

### NPCメモリシステム

```
短期：最後の3回の相互作用（ immediate context）
    ↓
中期：重要なイベント（関係の変化）
    ↓
長期：基本的な記憶（形成的な経験）
    ↓
意味的：埋め込みベースの想起（文脈的メモリ）
```

**メモリの影響：**
- 対話のトーンと語彙
- 信頼レベルと助ける意欲
- 挨拶スタイル（暖かい、冷たい、怖い）
- トピックの可用性（個人的、派閥、クエスト）

### 経済的・社会的行動

NPCは、歴史的パターンに基づいた**現実的な経済行動**を持っています：

| 行動 | ソース | 実装 |
|------|--------|------|
| **取引** | 中世の商人ギルド | 供給/需要、評判ベースの価格設定 |
| **職人技** | 歴史的な職人の工房 | スキルレベル、素材品質、時間投資 |
| **社会的力学** | 封建的階層 | 領主/家臣の関係、派閥忠誠 |
| **政治的陰謀** | 宮廷政治 | 秘密の同盟、情報取引 |

## ストーリー転換メカニクス

### 物語パターン選択

プレイヤーが行動を起こすと、システムは：

1. **意図を分析** — プレイヤーは何をしようとしているか？
2. **結果をシミュレート** — 現実的に何が起こるか？
3. **アーキタイプを選択** — どの聖書/文学パターンが適合するか？
4. **スタイルを適用** — どの文学的時代/雰囲気が対応するか？
5. **散文を生成** — パターン + スタイル + コンテキストを組み合わせる

### 例：プレイヤーが味方を裏切る

```
意図：裏切り
シミュレーション：関係破壊、派閥緊張上昇
アーキタイプ：創世記37（兄弟に売られたヨセフ）
スタイル：ヴィクトリア朝社会ドラマ
結果：ディケンズ的な社会的詳細で裏切りの結果を探る物語
```

### 動的難易度

クエストテンプレートには**道曖昧さスコア**（0-1）が含まれます：
- 0.0 — 明確な善悪（子供の物語）
- 0.5 — 複雑な選択（標準RPG）
- 1.0 — 正解なし（成熟した物語）

## パフォーマンスアーキテクチャ

### 前処理が勝つ理由

| アプローチ | レイテンシ | トークンコスト | 品質 |
|-----------|-----------|---------------|------|
| **オンザフライLLM** | 2-5秒 | 高い | 変動 |
| **前処理済みDB** | <100ms | 最小限 | 一貫した |
| **ハイブリッド（TNS）** | <100ms + ポリッシュ | 低い | 高い |

### ハイブリッドアプローチ

1. **DBクエリ** — 構造化テンプレートを取得（ミリ秒）
2. **変数置換** — コンテキストを埋める（マイクロ秒）
3. **スタイル適用** — 文学パターンを適用（ミリ秒）
4. **LLMポリッシュ** — 最終散文生成（オプション、重要な瞬間のため）

これにより、重要な場所で**LLM品質**を備えた**データベース速度**が得られます。

## 世界のルールと経済

### 封建的階層

TrueNeverStoryは、富、税金、特権、社会的相互作用を支配する**10段階の封建的ランクシステム**を実装しています：

| ランク | 最小富 | 警備兵 | 基本税率 | 給料 | 賄賂を渡せる | 賄賂を受け取れる |
|--------|--------|--------|----------|------|-------------|-----------------|
| **奴隷** | 0 | 0 | 100% | 0 | いいえ | いいえ |
| **平民** | 0 | 0 | 90% | 0 | はい | いいえ |
| **准男爵** | 100,000 | 50 | 30% | 0 | はい | はい |
| **男爵** | 500,000 | 200 | 28% | 0 | はい | はい |
| **子爵** | 2,000,000 | 1,000 | 25% | 0 | はい | はい |
| **伯爵** | 10,000,000 | 5,000 | 22% | 0 | はい | はい |
| **侯爵** | 50,000,000 | 20,000 | 20% | 0 | はい | はい |
| **公爵** | 200,000,000 | 100,000 | 18% | 0 | はい | はい |
| **王** | 1,000,000,000 | 500,000 | 15% | 0 | はい | はい |
| **皇帝** | 10,000,000,000 | 2,000,000 | 10% | 0 | はい | はい |

**重要なルール：**
- **奴隷**は賄賂経済に参加できない（自由意志がない）
- **平民**は賄賂を渡せるが受け取れない（権力がない）
- **上位ランク**は税金が低い（権力割引）が維持費が高い
- **富のしきい値**を達成して昇進する必要がある

### 税制

税制は**動的**で、複数の要因に影響されます：

```
実効税率 = 基本税率 × (1 - 権力割引 - 人気割引)
```

**コンポーネント：**
- **基本税率** — ランクで定義（平民90%、皇帝10%）
- **権力割引** — 政治的権力に基づく最大90%の削減（権力 / 10,000）
- **人気割引** — 公的承認に基づく最大30%の削減（人気 / 3,000）

**税の流れ：**
```
NPC収入 → 税計算 → 国庫徴収 → 領主の国庫
     ↓          ↓          ↓            ↓
  ランク      動的       ターンごと    封建的チェーン
  ベース      率         処理         沿っての蓄積
  給料                   忠誠チェック
```

**高税の結果：**
- **裏切りリスク** = (税負担 + 賄賂) / 収入 × (1 - 忠誠 / 1000)
- 高い税負担は反乱の可能性を高める
- 忠誠は裏切りに対する緩衝材として機能する

### 賄賂経済

賄賂は、政治、忠誠、権力のダイナミクスに影響を与える**第一級の経済メカニクス**です：

**賄賂の種類：**
| 種類 | 目的 | リスクレベル |
|------|------|-------------|
| **保護** | 罰則や迫害を避ける | 中 |
| **好意** | 優遇措置を得る | 低 |
| **沈黙** | 秘密を隠す | 高 |
| **アクセス** | 制限されたエリアや人に到達する | 中 |
| **昇進** | ランクや地位を上げる | 高 |
| **免除** | 税金や義務を避ける | 非常に高い |

**賄賂リスク式：**
```
リスク = 基本リスク (10%) + 金額リスク (金額 / 10,000) + 目撃者リスク (目撃者 × 15%) - 受取人のスキル (謀略 × 0.1%)
```

**リスク要因：**
- **金額** — 大きな賄賂はリスクが高い
- **目撃者** — 多くの観察者は発見のチャンスを高める
- **受取人の謀略** — スキルのある役人は腐敗を隠せる
- **賄賂の種類** — 一部の種類は本質的にリスクが高い

**経済的影響：**
- 賄賂は**忠誠を低下させる**（腐敗は信頼を侵食する）
- 賄賂は**受取人の富を増加させる**
- 賄賂は**権力を増加させる**（賄賂額の1%が権力に変換）
- 賄賂は**人気を低下させる**（公衆は腐敗を非難する）

### 経済サイクル（ヨセフモデル）

**聖書のヨセフのモデル**（創世記41）に基づき、経済は3つのフェーズを循環します：

**フェーズサイクル：**
```
豊穣 (30日) → 遷移 (10日) → 飢饉 (20日) → 豊穣...
```

**フェーズ効果：**
| フェーズ | 価格修正子 | 儲備変化 | 物語イベント |
|----------|-----------|----------|-------------|
| **豊穣** | 0.8× (安い) | ターンごと+20% | 収穫、交易ブーム |
| **遷移** | 1.0× (普通) | 安定 | 市場の不確実性 |
| **飢饉** | 2.0× (高い) | ターンごと-30% | 干ばつ、疫病、戦争 |

**戦略的意味：**
- **飢饉の生存のために**豊穣中に備蓄する
- **価格計画**はフェーズ遷移の予測が必要
- **プレイヤーの決定**はフェーズの期間と深刻さに影響できる

### ユビリー制度

**50年ごとに**、**ユビリーイベント**が大規模な経済リセットを引き起こします：

**ユビリー効果：**
1. **債務帳消し** — すべての未払い債務が帳消しになる
2. **土地返還** — すべての抵当土地が所有者に返還される
3. **忠誠ブースト** — すべてのNPCに+30%の忠誠ボーナス（10日間持続）

**歴史的基盤：**
永久的な貧困を防ぎ、社会的流動性を維持した**聖書のユビリー**（レビ記25）に基づいています。

**戦略的意味：**
- **債務のタイミング** — ユビリー年に近い貸し付けを避ける
- **土地取得** — 一時的な所有は興味深いダイナミクスを生む
- **社会的安定** — ユビリーは極端な富の集中を防ぐ

### 派閥税ジレンマ

2つの派閥が相反する税請求を持つ場合、システムはプレイヤーに**道徳的ジレンマ**を生成します：

**ジレンマ生成：**
- 経済ターンごとに30%の確率
- ジレンマ間の最低30日クールダウン
- 税額は100から1,000ゴールドの範囲

**プレイヤーの選択：**
| 選択 | 派閥A忠誠 | 派閥B忠誠 | 評判 |
|------|----------|----------|------|
| **派閥Aに支払う** | +50 | -30 | 中立 |
| **派閥Bに支払う** | -30 | +50 | 中立 |
| **両方拒否** | -20 | -20 | -10 (信頼できない) |

**物語統合：**
各ジレンマは紛争を説明するコンテキストストーリーを生成し、プレイヤーに意味のある政治的選択を強制します。

### 派閥労働規則

各派閥は、賃金と忠誠に影響する**個別の労働政策**を設定できます：

**労働規則パラメータ：**
- **固定賃金** — はい/いいえ（予測可能 vs. パフォーマンスベース）
- **賃金額** — 労働期間あたりの基本給
- **忠誠修正子** — 労働者忠誠へのボーナス/ペナルティ

**賃金計算：**
```
最終賃金 = 基本賃金 × 労働時間 × 生産性 × 派閥修正子
```

**忠誠の葛藤：**
NPCが複数の派閥で働く場合、忠誠の葛藤が生じる可能性があります：
- **二重忠誠**は効率を低下させる
- **派閥変更**には評判コストがある
- **労働停止**は紛争中に発生する

### 奴隷経済

歴史的な権力のダイナミクスを反映した**道徳的に複雑な**システム：

**奴隷の特性：**
- **健康** — 身体的状態（価値に影響）
- **経験** — スキルと知識（価値を増加）
- **悪徳** — 強欲、怒り、怠惰（価値を減少）
- **家族** — 奴隷は家族を持つことができる（義務を生む）

**奴隷価値式：**
```
価値 = 100 + (経験 × 0.1) + (健康 × 0.05) - 悪徳ペナルティ
```

**奴隷のライフサイクル：**
1. **奴隷化** — 借金、捕獲、または出生による
2. **労働** — 財産を生産（ターンあたり300-1000単位）
3. **反乱** — 警備兵が弱い場合に可能（強度比が成功を決定）
4. **解放** — 200ゴールドかかり、平民ステータスを付与

**倫理的考慮：**
- システムは**道徳的に不快**になるよう設計されている
- プレイヤーの選択は奴隷NPCに**実際の影響**を持つ
- **反乱メカニクス**は奴隷制が決して「安全」でないことを保証する
- **解放パス**は道徳的贖罪の機会を提供する

### ソーシャルグラフとの統合

経済は、現実的な権力のダイナミクスのために**ソーシャルグラフ**と統合されます：

**封建的関係：**
```
家臣 → 領主 (税 + 軍事義務)
  ↓
指揮系統 (複数レベル)
  ↓
税フロー蓄積 (徴収された税)
```

**派閥経済：**
- **経済派閥**は貿易と資源を制御する
- **軍事派閥**はセキュリティを提供する（対価として）
- **宗教派閥**は道徳的選択に影響する
- **犯罪派閥**は合法経済の外で活動する

**評判システム：**
- **公的承認**は税率に影響する
- **派閥の立場**は資源へのアクセスを決定する
- **個人的評判**は賄賂の成功率に影響する

### 経済データベース

すべての経済データは専用の**SQLiteデータベース**（`economic.db`）に保存されます：

**テーブル：**
- `economic_cycles` — 価格修正子付きフェーズ追跡
- `jubilee_events` — ユビリーの歴史的記録
- `faction_labor_rules` — 派派閥ごとの賃金政策
- `faction_dilemmas` — 税紛争の履歴

**パフォーマンス：**
- **インデックス付きクエリ**による高速経済計算
- **バッチ処理**によるNPC操作（年齢減衰、悪徳減衰、税、忠誠）
- **Mojoカーネル**による計算集約的操作（5×高速化）

## まとめ

TrueNeverStoryのアーキテクチャは以下を組み合わせています：

1. **LLM品質とトークン効率のための英語優先処理**
2. **豊かな物語パターンへの即時アクセスのための前処理済み文学データベース**
3. **信じられ、記憶に残るキャラクターのための多層NPCシステム**
4. **速度と品質のバランスを取るハイブリッド生成**

その結果、**データベース速度**で**文学的品質の物語**を生成しながら、**深いキャラクターの一貫性**と**意味のあるプレイヤーの選択**を維持するシステムとなっています。
